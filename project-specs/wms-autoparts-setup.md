Техническое задание: Внутренняя система управления заказами на автозапчасти (Заказ-Справочник)
1. Назначение
Создать локальное веб-приложение для формирования заказов автозапчастей механиками, передачи заказов на склад и управления справочником запчастей. Система ориентирована на работу в локальной сети организации.

2. Технологический стек
Frontend/Backend: Next.js 14+ (App Router)

UI библиотека: MUI (Material-UI)

API: Next.js API Routes

Анализ кода: ESLint

ORM: Drizzle ORM

База данных: PostgreSQL

Контейнеризация: Docker (отдельные контейнеры для приложения и БД)

Аутентификация: next-auth (с поддержкой ролей)

Генерация PDF: библиотека на выбор (например, @react-pdf/renderer или jsPDF)

Загрузка файлов: локальное хранилище (папка public/uploads)

Мультиязычность: next-intl (русский, английский, арабский)

Адаптивность: только десктоп (минимальное разрешение 1280x720)

3. Функциональные требования
3.1. Аутентификация и авторизация
Регистрация доступна только для первого пользователя (становится администратором, не удаляется).

Остальные пользователи создаются администратором через панель управления.

Авторизация на основе ролей (см. раздел «Роли»).

Сессии хранятся в базе данных (next-auth + PostgreSQL).

3.2. Роли и права доступа
Базовая система ролей (создаётся при инициализации БД):

MECHANIC – просмотр каталога, создание заявок (черновиков), редактирование своих черновиков, просмотр своих заявок.

REPAIR_MANAGER – все права механика + просмотр всех заявок своего отдела, согласование/отклонение, назначение приоритета.

STOREKEEPER – просмотр согласованных заявок, отметка о выдаче, управление остатками (приём поставок, списание), просмотр истории остатков.

ADMIN – полный доступ ко всем функциям, управление пользователями, управление справочниками (иерархия, запчасти, типы узлов), управление ролями.

Гибкость ролей:

Администратор может создавать новые роли (например, «главный кладовщик» с расширенными правами), комбинируя разрешения.

Разрешения хранятся в виде JSON в таблице RoleType. При создании роли администратор выбирает разрешения из предопределённого списка (например, view_all_orders, edit_stock, manage_users и т.д.).

Разрешения (примерный набор):

catalog_view – просмотр каталога

order_create – создание заказа

order_edit_own_draft – редактирование своих черновиков

order_view_own – просмотр своих заказов

order_view_all – просмотр всех заказов

order_approve – согласование заказов

order_fulfill – выдача запчастей

stock_manage – управление остатками (приём, списание)

stock_view_history – просмотр истории остатков

user_manage – управление пользователями

role_manage – управление ролями

hierarchy_manage – управление иерархией узлов

parts_manage – управление запчастями

reports_view – просмотр аналитики

settings_access – доступ к настройкам

3.3. Работа с иерархией запчастей
Гибкая древовидная структура (Materialized Path) с корневым узлом root (не отображается в интерфейсе).

Типы узлов (NodeType) предопределены, но администратор может создавать новые (например, добавить уровень «Тип двигателя»).

Для каждого узла можно задать атрибуты (JSON), которые отображаются в интерфейсе (опционально).

Администратор может:

создавать, редактировать, удалять узлы;

перемещать узлы в другое место иерархии;

изменять тип узла;

управлять атрибутами узла.

3.4. Управление запчастями
Каждая запчасть привязана к одному узлу-детали (node_type = 'part' или аналогичному).

Поля запчасти: название, артикул, описание, цена (опционально), остаток на складе (целое число), минимальный остаток, место хранения (строка), изображения (до нескольких, одно основное).

Администратор может добавлять, редактировать, удалять запчасти.

Для запчасти могут быть заданы спецификации (JSON), структуру которых определяет администратор (например, вес, материал, производитель). Поле может быть пустым.

3.5. Управление заказами
Статусная модель (хранится в справочнике OrderStatus):

draft – черновик (механик может редактировать)

submitted – отправлен на согласование (редактирование запрещено)

approved – согласован (ожидает выдачи)

rejected – отклонён (с комментарием)

partially_fulfilled – выдан частично

fulfilled – выдан полностью

Переходы:

Черновик → Отправлен (механик)

Отправлен → Согласован / Отклонён (repair_manager)

Согласован → Выдан частично / Выдан полностью (storekeeper)

При частичной выдаче остаётся возможность довыдачи (до полного выполнения).

Приоритеты: высокий (1), средний (2), низкий (3). Назначает repair_manager.

Поля заказа:

механик (кто создал)

repair_manager (кто согласовал, опционально)

статус, приоритет, примечания

даты создания, согласования, выполнения

позиции заказа (запчасть, количество, количество выданное)

Действия:

Механик создаёт черновик, добавляет позиции, может удалять позиции, изменять количество, сохранять черновик.

Отправка черновика делает его недоступным для редактирования.

Repair_manager видит список отправленных заявок, может просмотреть детали, согласовать или отклонить с комментарием.

Storekeeper видит согласованные заявки, отмечает факт выдачи (полностью или частично). При выдаче автоматически уменьшаются остатки на складе и создаётся запись в истории.

3.6. Управление складом
Кладовщик может просматривать текущие остатки по всем запчастям.

Кладовщик может принять новую поставку: выбрать запчасть, указать количество, причину (приход), при необходимости добавить комментарий. Остатки увеличиваются, запись в истории.

Кладовщик может списать запчасти (например, при браке) с указанием причины.

История изменений остатков хранится в таблице StockHistory с указанием пользователя, количества, причины, связанного заказа (если выдача) и комментария.

3.7. Интерфейс (главная страница)
Макет (одна страница с тремя зонами):

Левая панель – иерархическое дерево категорий (Materialized Path).

Поддержка ленивой загрузки (при раскрытии узла загружаются его дочерние элементы).

Отображение количества запчастей в узле (если есть).

Возможность развернуть/свернуть всё.

Текущий узел подсвечивается.

Центральная область – список запчастей выбранного узла.

Отображение: плитка (по умолчанию) с изображением, названием, артикулом, остатком, ценой (опционально).

Возможность переключения на табличный вид.

Поиск (глобальный) – см. ниже.

Для каждой запчасти кнопка «Добавить в заказ».

Верхняя панель – хлебные крошки, поле глобального поиска, кнопка «Создать заказ», переключение языка, мини-корзина (отображает количество позиций, при клике – выпадающий список с возможностью изменить количество/удалить).

Поиск:

Глобальный по всем запчастям и узлам.

Результаты группируются: сначала запчасти, затем узлы.

При клике на результат (запчасть) открывается узел, содержащий эту запчасть, и запчасть подсвечивается; при клике на узел – открывается этот узел в дереве.

Корзина:

Хранится в состоянии (Zustand) до момента оформления заказа.

При добавлении запчасти открывается диалог выбора количества.

В мини-корзине можно изменять количество, удалять позиции.

Кнопка «Оформить заказ» создаёт черновик (если у пользователя есть право на создание заказов) и очищает корзину.

Если пользователь не авторизован, корзина не сохраняется между сессиями.

Создание заказа:

Из корзины → создаётся черновик.

Из черновика можно перейти на страницу редактирования заказа (отдельная страница), где можно менять состав, количество, добавить комментарий.

Отправка черновика на согласование.

3.8. Панель управления (администрирование)
Доступна только для пользователей с соответствующими правами (admin, а также возможно для других ролей с разрешениями).

Разделы (левое меню):

Аналитика – минимальные показатели: количество заказов по статусам, популярные запчасти, остатки ниже минимального уровня. В будущем легко расширять.

Каталог:

Иерархия – управление узлами (CRUD, перемещение, типы узлов).

Запчасти – управление запчастями (CRUD, загрузка изображений).

Типы узлов – управление NodeType (можно добавлять/редактировать, менять иконки, default_attributes).

Склад:

Остатки – просмотр и корректировка остатков (приход/списание).

История – просмотр всех изменений остатков.

Поставщики – (заглушка, для будущего расширения).

Пользователи:

Список пользователей – создание, редактирование, блокировка/удаление (кроме первого администратора).

Роли и права – управление ролями (создание, редактирование разрешений).

Настройки (опционально) – настройки системы (языки по умолчанию, параметры PDF и т.п.).

3.9. Генерация PDF
PDF формируется для одобренного заказа (статус approved).

Содержит: номер заказа, дату, данные механика, список запчастей (артикул, название, количество, место хранения), итоговое количество позиций, подписи (механик, кладовщик) – можно текстовые поля для печати.

Доступна кнопка «Печать»/«Скачать PDF» на странице просмотра заказа.

Гибкость: структура PDF может быть изменена в будущем (добавление штрихкодов, расширенных данных). Желательно вынести шаблон в отдельный компонент.

3.10. Мультиязычность
Используется next-intl.

Языки: русский (RU), английский (EN), арабский (AR – Ливан).

Файлы переводов структурированы по категориям: common, catalog, orders, admin, errors и т.д.

Переключение языка – в шапке, сохраняется в cookie.

4. Структура базы данных (предварительная, может уточняться в процессе)
sql
-- Роли (справочник)
CREATE TABLE role_types (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL, -- mechanic, repair_manager, storekeeper, admin
    display_name VARCHAR(100),
    permissions JSONB NOT NULL DEFAULT '[]',
    sort_order INTEGER,
    is_system BOOLEAN DEFAULT FALSE
);

-- Статусы заказов (справочник)
CREATE TABLE order_statuses (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL,
    display_name VARCHAR(100),
    color VARCHAR(20),
    is_final BOOLEAN DEFAULT FALSE,
    is_editable BOOLEAN DEFAULT FALSE,
    allowed_transitions INTEGER[] -- массив id статусов, в которые можно перейти
);

-- Типы узлов иерархии
CREATE TABLE node_types (
    id SERIAL PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL,
    display_name VARCHAR(100),
    icon VARCHAR(50),
    default_attributes JSONB,
    can_have_children BOOLEAN DEFAULT TRUE,
    can_have_parts BOOLEAN DEFAULT FALSE,
    sort_order INTEGER
);

-- Пользователи
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    full_name VARCHAR(255),
    role_type_id INTEGER REFERENCES role_types(id) ON DELETE RESTRICT,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW()
);

-- Иерархия узлов (Materialized Path)
CREATE TABLE item_hierarchy (
    id SERIAL PRIMARY KEY,
    parent_id INTEGER REFERENCES item_hierarchy(id) ON DELETE CASCADE,
    node_type_id INTEGER REFERENCES node_types(id),
    name VARCHAR(255) NOT NULL,
    path VARCHAR(1024) NOT NULL, -- например, "root.1.5.23"
    sort_order INTEGER,
    attributes JSONB
);

-- Запчасти
CREATE TABLE parts (
    id SERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    part_number VARCHAR(100) UNIQUE NOT NULL,
    description TEXT,
    stock INTEGER NOT NULL DEFAULT 0,
    price DECIMAL(10,2),
    min_stock_level INTEGER DEFAULT 0,
    location VARCHAR(100),
    hierarchy_id INTEGER REFERENCES item_hierarchy(id) ON DELETE RESTRICT,
    specifications JSONB
);

-- Изображения запчастей
CREATE TABLE part_images (
    id SERIAL PRIMARY KEY,
    part_id INTEGER REFERENCES parts(id) ON DELETE CASCADE,
    image_url VARCHAR(500) NOT NULL,
    is_primary BOOLEAN DEFAULT FALSE
);

-- Заказы
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    mechanic_id INTEGER REFERENCES users(id) ON DELETE RESTRICT,
    repair_manager_id INTEGER REFERENCES users(id),
    status_id INTEGER REFERENCES order_statuses(id),
    priority INTEGER CHECK (priority IN (1,2,3)), -- 1-высокий, 2-средний, 3-низкий
    notes TEXT,
    approved_at TIMESTAMP,
    completed_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Позиции заказа
CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INTEGER REFERENCES orders(id) ON DELETE CASCADE,
    part_id INTEGER REFERENCES parts(id) ON DELETE RESTRICT,
    quantity INTEGER NOT NULL CHECK (quantity > 0),
    quantity_fulfilled INTEGER DEFAULT 0,
    status VARCHAR(20) DEFAULT 'pending' -- pending, partially_fulfilled, fulfilled
);

-- История изменения остатков
CREATE TABLE stock_history (
    id SERIAL PRIMARY KEY,
    part_id INTEGER REFERENCES parts(id) ON DELETE CASCADE,
    user_id INTEGER REFERENCES users(id),
    quantity_change INTEGER NOT NULL, -- положительное или отрицательное
    reason VARCHAR(100) NOT NULL, -- order_fulfillment, stock_receipt, manual_correction, etc.
    order_id INTEGER REFERENCES orders(id) ON DELETE SET NULL,
    notes TEXT,
    created_at TIMESTAMP DEFAULT NOW()
);
Примечания по БД:

Корневой узел root создаётся при инициализации БД (parent_id NULL, path = 'root').

Для поля permissions в role_types рекомендуется использовать массив строк, например ["catalog_view", "order_create"].

Для статусов заказов allowed_transitions содержит массив id статусов, в которые можно перейти из текущего.

6. Нефункциональные требования
Производительность: загрузка списка запчастей для узла не более 500 мс, ленивая загрузка дерева.

Безопасность: пароли хешируются (bcrypt), all API проверяют сессию и права.

Надёжность: при изменении остатков используется транзакция.

Логирование: ошибки записываются в консоль (для локальной системы достаточно).

7. Деплой
Приложение упаковывается в Docker-образ (Dockerfile).

База данных – отдельный контейнер PostgreSQL.

Запуск через docker-compose (два сервиса: app, postgres).

Переменные окружения (база данных, секреты) передаются через .env.

8. Комментарии по гибкости
Схема БД может быть изменена в процессе разработки. Описанная структура является отправной точкой.

Для статусов заказов и ролей предусмотрены справочники, позволяющие добавлять новые значения без изменения кода.

Разрешения ролей хранятся в JSON, что позволяет гибко настраивать права.

PDF-генерация вынесена в отдельный сервис, чтобы легко менять шаблон.

Аналитика реализована минимально, но архитектура позволяет расширять.

## 9. Тестирование

### 9.1. Стратегия тестирования
- **Unit-тесты** – для сервисного слоя (бизнес-логика), утилит, валидации.
- **Интеграционные тесты** – для API эндпоинтов (с реальной БД в тестовом окружении).
- **E2E-тесты** – для критических пользовательских сценариев.

### 9.2. Инструменты
- **Jest** + **React Testing Library** – для unit и интеграционных тестов.
- **Supertest** или **next-test-api-route** – для API.
- **Playwright** – для E2E-тестов.
- **Drizzle** – для работы с тестовой БД (использовать транзакции с откатом).

### 9.3. Организация
- Тесты размещаются рядом с кодом (`__tests__` или `.test.ts`).
- Запуск: `npm test` (unit+integration) и `npm run test:e2e`.
- Целевое покрытие – не менее 70% для ключевых модулей.

### 9.4. Критические сценарии
- Аутентификация и авторизация (роли, доступы).
- Иерархия (CRUD, перемещение узлов, materialized path).
- Запчасти (CRUD, изображения, привязка к узлу).
- Корзина (состояние, добавление, изменение количества).
- Заказы (создание, статусные переходы, выдача).
- Склад (приход, списание, история).
- Управление ролями (создание роли, проверка разрешений).

### 9.5. CI/CD (опционально)
- При возможности – настройка GitHub Actions для автоматического запуска тестов.
